# Database Setup Instructions for Unified Maintenance Management

## Overview
The system now includes a **MaintenanceScheduler** that automatically generates maintenance plans from schedules. To enable this feature, the `pm_schedules` table needs to be created in Supabase.

## Required Tables

### 1. `pm_schedules` Table
This table stores maintenance scheduling rules that trigger automatic plan generation.

**SQL to create the table:**

```sql
CREATE TABLE pm_schedules (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  asset_id UUID NOT NULL REFERENCES assets(id) ON DELETE CASCADE,
  task_description TEXT NOT NULL,
  frequency_days INTEGER,
  threshold_value DECIMAL,
  categoria TEXT DEFAULT 'Preventiva',
  created_by UUID NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  is_active BOOLEAN DEFAULT true,
  last_execution_date TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Create indexes for better performance
CREATE INDEX pm_schedules_asset_id_idx ON pm_schedules(asset_id);
CREATE INDEX pm_schedules_is_active_idx ON pm_schedules(is_active);
CREATE INDEX pm_schedules_frequency_idx ON pm_schedules(frequency_days);
```

**Fields:**
- `id`: UUID, primary key
- `asset_id`: Reference to the asset this schedule applies to
- `task_description`: Description of the maintenance task
- `frequency_days`: Number of days between maintenance (for calendar-based schedules)
- `threshold_value`: Usage threshold that triggers maintenance (for threshold-based schedules)
- `categoria`: Type of maintenance (e.g., "Preventiva", "Corretiva")
- `created_by`: User ID who created this schedule
- `created_at`: Timestamp when created
- `is_active`: Boolean flag to enable/disable scheduling
- `last_execution_date`: Last time a plan was auto-generated from this schedule
- `updated_at`: Last modification timestamp

### 2. `maintenance_plans` Table (Already Exists)
The system uses this existing table for maintenance execution plans. When a schedule triggers, a new maintenance_plan is created.

**Expected fields in `maintenance_plans`:**
- `id`: UUID primary key
- `asset_id`: Reference to asset
- `tipo`: Type of maintenance
- `periodicidade`: Frequency description
- `proxima_data`: Next scheduled date
- `responsavel_id`: User responsible for execution
- `custo_estimado`: Estimated cost
- `descricao`: Description
- `estado` or `status`: Current state ("Agendado", "Em Progresso", "Concluído", etc.)
- `data_conclusao` or `completed_at`: Completion timestamp
- `observacoes` or `observations`: Work notes
- `auto_generated`: Flag indicating if auto-generated by scheduler (new field, optional)

## Scheduler Behavior

The **MaintenanceScheduler** runs every hour and:

1. Fetches all active schedules from `pm_schedules` where `is_active = true`
2. For each schedule:
   - Checks if `frequency_days` has elapsed since `last_execution_date`
   - If YES: Creates a new `maintenance_plan` record
   - Updates `last_execution_date` to current time
   - Creates a notification for the schedule creator
3. Logs all activity to `logs/server.log`

## New Unified API Endpoints

### Schedules Management
- **GET** `/api/maintenance/schedules` - List all maintenance schedules
- **POST** `/api/maintenance/schedules` - Create new schedule
  ```json
  {
    "asset_id": "uuid",
    "task_description": "Maintenance description",
    "frequency_days": 30,
    "categoria": "Preventiva"
  }
  ```

### Active Plans Management
- **GET** `/api/maintenance/executions` - List active maintenance plans
- **PATCH** `/api/maintenance/executions/:id/start` - Mark plan as in progress
- **PATCH** `/api/maintenance/executions/:id/complete` - Mark plan as complete
  ```json
  {
    "observations": "Work completed successfully",
    "actual_cost": 150.50
  }
  ```

### History & Analytics
- **GET** `/api/maintenance/history` - List completed maintenance plans with metrics

## Implementation Steps

1. **Create `pm_schedules` table** in Supabase using the SQL above
2. **Wait for MaintenanceScheduler** to start (runs on server startup)
3. **Monitor logs** in `logs/server.log` for scheduler activity
4. **Create a schedule** via POST `/api/maintenance/schedules`
5. **Wait up to 1 hour** or trigger manually for scheduler to generate plan
6. **Verify plan creation** in maintenance_plans table and UI

## Troubleshooting

### "Tabela 'pm_schedules' não encontrada" Warning
- Create the `pm_schedules` table using SQL above
- Restart the server
- The warning will disappear on next scheduler run

### Plans not auto-generating
- Check that schedule has `is_active = true`
- Verify `frequency_days` is set and greater than days elapsed
- Check `logs/server.log` for scheduler errors
- Ensure `last_execution_date` is null or older than `frequency_days` ago

### Notifications not created
- Verify `notifications` table exists
- Check that schedule `created_by` user ID is valid
- Review error logs for database write errors
